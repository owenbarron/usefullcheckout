Identity model

How Stripe fingerprinting helps

1. Physical card lookup

Stripe Terminal returns a fingerprint token for the tapped card. We can look up the saved account from that token without USEFULL ever handling the raw card number.

2. Wallet-to-card rough match

Apple Pay / Google Pay usually surface the real card's last4 even though the wallet uses a different fingerprint. That lets us roughly link wallet taps to the same underlying physical card.

3. Known exception

Apple Card added to Apple Pay can surface only the wallet number. In those cases, or any time card details don't match, we step up to OTP verification.

4. Hard fallback

Account identifier remains the durable fallback. If the tapped card details don't line up with the account, verify ownership with OTP before checkout.

Practical rule: fingerprint is the zero-friction path, card-details matching is the low-friction recovery path, and OTP is the trust-preserving fallback.

Fee model

Saved card. Existing fee logic.

Save the card at checkout via Stripe SetupIntent — no charge. Fees work exactly like they do today: daily accrual, weekly batched Stripe charges, same backend, same infrastructure.

Zero changes to the charging system. The kiosk just adds a new way to save a card.

One UX requirement: Stripe requires disclosure at the kiosk that the card will be saved for future charges. This is a copy/design task, not a technical blocker.

Evaluated & ruled out

Why not a small deposit?

The $0.50 deposit idea doesn't hold up.

12%+

Effective processing rate. ~6¢ on a 50¢ charge — the fee eats the deposit.

5–10 days

For refunds to appear. Students return the container, charge lingers on their statement.

No behavior change

Loss-aversion threshold is ~$5+. A 50¢ deposit doesn't move the needle.

Confusing statements

Charge + refund + later fee = multiple line items = support tickets and disputes.

The saved-card model accomplishes the same accountability without the cost or confusion.

Also evaluated

Why not these approaches?

×

Square

SDK explicitly prohibits unattended terminals and kiosks. Disqualified.

×

Accountless model

Fee enforcement requires a saved card. "Accountless" still saves a card under the hood — it's architecturally identical to account-based. Ship account-based.

×

Full hold, void on return

Works for one-time visitors (Holy Cross), but doesn't preserve our graduated fee schedule. Boulder is a recurring campus program — our existing fee logic is a better fit.

Holy Cross

Clover integration — stadium concessions

Different problem, different hardware, same USEFULL backend.

Checkout

  • Clover-native app on existing concession POS
  • Cashier scans container QR, places pre-auth hold
  • No separate USEFULL checkout kiosk needed

Returns

  • USEFULL return stations: ATR200 + tablet
  • Full hold ($17/$25), void on return within 24 hrs
  • Right model for one-time stadium visitors

Note: Clover runs on Fiserv, not Stripe — separate integration path. State grant opportunity has resurrected this project.
